# ADR 0036: Stale Context Validation for Click Actions

## Status

Accepted

## Date

2026-06-04

## Context

BrowserBridge uses short-lived positional IDs (`bb-1`, `bb-2`, etc.) to identify elements on the page. These IDs are generated by `read_current_page` and reference elements by their position in the DOM (e.g., `bb-3` means "the 3rd visible `<a href>` link on the page").

The current contract (ADR 0012, ADR 0013) expects agents to call `read_current_page` then immediately `click_element` with fresh IDs. In practice, agents — especially multi-step LLM agents — naturally read once and then act multiple times. Between the read and the click:

- The page may navigate (full page load)
- React/SPA may re-render (DOM changes without navigation)
- Lazy content may load, shifting element positions
- Ads or modals may inject elements, pushing indexes

When IDs go stale, `click_element` silently clicks the **wrong element** — or succeeds on the wrong target and the agent believes it acted correctly. This is worse than an error because the agent has no signal to re-read and retry.

Today's `performClick` in the content script simply finds the element at position N and calls `.click()`. It has no way to know whether element N is the same element the agent intended. The MCP tool returns `{ ok: true }` regardless of which element was actually clicked.

This affects all action tools (`fill_input`, `fill_editable`, `set_checked`, `select_options`, `submit_form`) that use positional IDs, but **click is the highest-risk action** because it navigates or triggers side effects.

## Decision

Add **optional validation fields** to action targets so the content script can verify the element at the given position matches what the agent intended. If validation fails, return a structured `stale_context` error instead of silently clicking the wrong thing.

### 1. Extend `ClickActionTarget` with optional validation fields

```ts
export interface ClickActionTarget {
  kind: "link" | "action";
  id: string;
  // Optional validation — if provided, the content script verifies
  // the element at position id matches before clicking.
  expectedText?: string;
  expectedHref?: string; // only for kind: 'link'
  expectedRole?: string; // only for kind: 'action' — e.g. 'button', 'tab'
}
```

All validation fields are optional. When omitted, the behavior is identical to today — find by position, click without validation. This preserves backward compatibility and keeps simple use cases simple.

### 2. Validation logic in `performClick`

When validation fields are present, `findClickTarget` resolves the element as before, then checks:

- `expectedText`: The element's visible text content (trimmed) must contain the expected text as a substring (case-insensitive). This is fuzzy because page context often truncates long link text.
- `expectedHref`: For `kind: 'link'`, the element's `href` attribute must match the expected path. Comparison is pathname-only (ignores query strings and fragments) to handle SPA routing.
- `expectedRole`: For `kind: 'action'`, the element's `role` attribute or implicit role must match.

If any validation check fails, `performClick` returns:

```ts
{
  ok: false,
  error: {
    code: 'stale_context',
    message: 'Element at position bb-3 does not match expected target. '
           + 'Expected text containing "My Jobs", found "Payments History". '
           + 'Call read_current_page and retry with fresh IDs.',
    detail: {
      id: 'bb-3',
      kind: 'link',
      expectedText: 'My Jobs',
      foundText: 'Payments History',
      foundHref: '/portal/payments'
    }
  }
}
```

The error includes enough context for the agent to understand what happened and what to do next.

### 3. Extend MCP tool schema

The `click_element` tool input schema gains optional fields:

```json
{
  "kind": { "type": "string", "enum": ["link", "action"] },
  "id": { "type": "string" },
  "browserInstanceId": { "type": "string" },
  "expectedText": {
    "type": "string",
    "description": "Optional: validate element text before clicking"
  },
  "expectedHref": {
    "type": "string",
    "description": "Optional: for links, validate href before clicking"
  },
  "expectedRole": {
    "type": "string",
    "description": "Optional: for actions, validate role before clicking"
  }
}
```

### 4. Apply same pattern to other action tools

The same validation approach should apply to `fill_input`, `fill_editable`, `set_checked`, `select_options`, and `submit_form`. Each gains optional `expectedLabel` or `expectedText` fields. Implementation can be phased — click first, then other tools.

### 5. Update skill guidance

The BrowserBridge MCP skill and `using-browserbridge` skill should document:

- Always pass `expectedText` (and `expectedHref` for links) when calling action tools after a page read
- A `stale_context` error means re-read the page and retry — do not retry with the same IDs
- When `expectedText` is empty or unknown, omit it — this is valid for dynamic content

## Alternatives Considered

### A. Auto-re-read on stale detection

Instead of returning an error, automatically re-read the page, find the matching element by text/href, and click it. **Rejected**: Auto-re-read violates the security model (the agent must explicitly request reads) and hides navigation from the agent. The agent should see the new page state and decide what to do.

### B. Durable element IDs (stable across DOM changes)

Assign IDs based on element identity (e.g., hash of href + text) rather than position. **Rejected**: Stable IDs are fragile in other ways — text changes, hrefs change across locales/A/B tests, and two elements can share the same text. Positional IDs plus validation give the best trade-off: simple, deterministic, with a safety net.

### C. Require re-read before every action

Mandate that agents must call `read_current_page` before any action, and the MCP server enforces this by attaching a context hash. **Rejected**: This adds latency (extra round-trip per action) and complexity (context hash tracking). Validation fields are a lighter-weight approach that achieves the same safety without mandatory re-reads.

### D. CSS selectors instead of positional IDs

Use full CSS selectors (e.g., `a[href="/portal/jobs"]`) instead of positional IDs. **Rejected**: CSS selectors are harder for agents to construct correctly, harder to validate, and don't handle dynamic class names in SPA frameworks. The current `read_current_page` → `click_element` flow is simpler for agents.

## Consequences

**Positive:**

- Agents get a clear `stale_context` error instead of silently clicking the wrong thing
- Backward compatible — omitting validation fields preserves current behavior
- Implementation is localized to content-handler.ts and protocol.ts types
- Error messages include enough detail (found vs expected) for the agent to self-correct

**Negative:**

- Slight increase in wire payload (optional fields per action)
- Agents (and MCP clients calling the tool) need to be updated to pass validation fields — existing callers that don't pass them will continue to work but won't get validation
- Fuzzy text matching (substring, case-insensitive) may produce false positives in rare cases with very short generic text like "View" — but this is acceptable because the agent can always pass a more specific `expectedText`

**Neutral:**

- The protocol version stays the same — this is a backwards-compatible addition
- Content script validation increases response time negligibly (string comparison on a single element)

## Implementation Plan

2. Update `performClick` and `findClickTarget` in `packages/shared/src/content-handler.ts` — add validation
3. Update the `click_element` tool schema in `servers/mcp/src/mcp-server.ts`
4. Add unit tests for stale-context detection in `packages/shared/src/content-handler.test.ts`
5. Add integration test for stale-context error propagation
6. Update the BrowserBridge skill documentation
