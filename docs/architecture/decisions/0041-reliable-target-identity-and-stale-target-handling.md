# ADR 0041: Reliable Target Identity and Stale-Target Handling

**Status:** Proposed
**Date:** 2026-06-09
**Supersedes:** ADR 0036 (extends stale-context validation to all action tools)

## Problem Context

BrowserBridge uses short-lived positional IDs (`bb-1`, `bb-2`, etc.) generated by `read_current_page`. ADR 0036 introduced optional validation fields (`expectedText`, `expectedHref`, `expectedRole`) for `click_element` — and this is now implemented. However, three critical gaps remain:

### Gap 1: Other action tools lack stale-context protection

`fill_input`, `fill_editable`, `set_checked`, `select_options`, and `submit_form` all use positional IDs (`formId`, `controlId`, `id`) but have **no validation fields**. The same stale-DOM problem that affects clicks also affects these tools — an agent reads the page once, then fills a form field, but by that time the form may have re-rendered and the agent fills the wrong input. Today this fails silently or with a confusing `target_not_found`.

### Gap 2: Post-navigation invalidation is invisible

After a click that triggers navigation, **every cached ID from the previous `read_current_page` is invalid** — not just the clicked element. The agent has no signal that its entire page-context snapshot is stale. Today, subsequent calls using old IDs may silently target wrong elements or return `target_not_found`, giving the agent no structured way to understand _why_ things broke.

### Gap 3: `target_not_found` conflates stale IDs with missing elements

When a positional ID no longer resolves, `target_not_found` is returned regardless of why. The agent cannot distinguish between "this element used to exist but the page changed" (stale — re-read) and "this element was never on this page" (bug — don't retry).

## Decision

### 1. Add validation fields to all action targets

Extend the same ADR 0036 validation pattern to every action tool target type. Each target gains an `expectedLabel` field (the visible label or text the agent expects), plus specialized fields per action type.

#### 1a. Form control targets (`fill_input`, `set_checked`, `select_options`)

```ts
export interface FormControlTarget {
  formId: string;
  controlId: string;
  /** Optional: validate the control's visible label/text matches before acting */
  expectedLabel?: string;
}
```

- `expectedLabel`: Matches the visible label or placeholder text of the control (substring, case-insensitive). Extracted from `<label>`, `aria-label`, `title`, or `placeholder`.

When validation fails, returns:

```ts
{
  ok: false,
  error: {
    code: 'stale_context',
    message: 'Form control bb-5 in form bb-1 does not match expected target. '
           + 'Expected label containing "Email", found "Password". '
           + 'Call read_current_page and retry with fresh IDs.',
    detail: {
      formId: 'bb-1',
      controlId: 'bb-5',
      expectedLabel: 'Email',
      foundLabel: 'Password',
      foundType: 'password'
    }
  }
}
```

#### 1b. Editable targets (`fill_editable`)

```ts
export interface EditableActionTarget {
  kind: "editable";
  id: string;
  /** Optional: validate the editable element's visible text/aria-label matches */
  expectedText?: string;
}
```

- `expectedText`: Matches the editable element's current text content or `aria-label` (substring, case-insensitive).

#### 1c. Form submit targets (`submit_form`)

```ts
export interface FormSubmitTarget {
  formId: string;
  /** Optional: validate the form contains an expected heading or landmark */
  expectedLabel?: string;
}
```

- `expectedLabel`: Matches a visible heading or label within the form (substring, case-insensitive). Catches the case where the DOM reorganized and `bb-1` now points to a different form.

### 2. Post-navigation page invalidation

Add a `page_context_id` (monotonic counter) to every `read_current_page` response. The content script increments this counter on every `pageshow` event (full navigation), `popstate` event (back/forward), and when it detects a significant DOM mutation (optional, heuristic).

When an action tool receives IDs from a `page_context_id` that no longer matches the current page's context ID, return:

```ts
{
  ok: false,
  error: {
    code: 'page_navigated',
    message: 'The page has navigated since the last read. '
           + 'Call read_current_page to get fresh context.',
    detail: {
      previousContextId: 3,
      currentContextId: 4
    }
  }
}
```

This is **separate from** `stale_context`. A `page_navigated` error is definitive: the entire previous snapshot is invalid. A `stale_context` error means the page might have shifted minimally — re-reading might produce similar IDs, or the agent might choose to omit validation and proceed anyway.

**Implementation:**

- Content script maintains a `pageContextVersion` counter starting at 1, stored in the content script's module scope.
- `read_current_page` includes `pageContextId: pageContextVersion` in its response under `data.context`.
- Every action request envelope that carries positional IDs (`perform_action` with `click`, `write_text`, `set_checked`, `select_options`, `submit_form`) includes an optional `pageContextId` field.
- The content script checks: if `pageContextId` is provided and doesn't match `pageContextVersion`, return `page_navigated` error before any target resolution.
- The content script increments `pageContextVersion` on every `pageshow` event (covers full navigations including back/forward cache). SPA-internal DOM mutations are NOT included — those are handled by `stale_context` validation.

### 3. Split `target_not_found` into distinct error codes

Today, `target_not_found` means "no element at that position". Split into:

| Error code         | Meaning                                                        | Agent behavior                    |
| ------------------ | -------------------------------------------------------------- | --------------------------------- |
| `stale_context`    | Element exists at position but doesn't match validation fields | Re-read and retry with fresh IDs  |
| `page_navigated`   | Page navigated since read; all IDs are invalid                 | Re-read; entire snapshot is stale |
| `target_not_found` | No element at that position at all (ID out of range)           | Likely a bug; re-read to verify   |

The distinction gives agents clear recovery guidance.

### 4. MCP tool schema changes

Each action tool gains its validation field plus `pageContextId`:

#### `click_element`

Already has `expectedText`, `expectedHref`, `expectedRole`. **Add**: `pageContextId` (optional integer).

#### `fill_input`

**Add**: `expectedLabel` (optional string), `pageContextId` (optional integer).

#### `fill_editable`

**Add**: `expectedText` (optional string), `pageContextId` (optional integer).

#### `set_checked`

**Add**: `expectedLabel` (optional string), `pageContextId` (optional integer).

#### `select_options`

**Add**: `expectedLabel` (optional string), `pageContextId` (optional integer).

#### `submit_form`

**Add**: `expectedLabel` (optional string), `pageContextId` (optional integer).

### 5. Update `read_current_page` response

Add `pageContextId` to the page context response:

```ts
// In the page context response data:
{
  context: {
    pageContextId: 3,    // <-- new field
    url: '...',
    title: '...',
    structure: { ... },
    // ...existing fields
  }
}
```

### 6. Protocol changes

#### Content script request types

Each `perform_action` sub-type gains `pageContextId`:

```ts
type PerformActionRequest = {
  type: "perform_action";
  pageContextId?: number; // <-- new optional field
  action:
    | PerformClickAction
    | PerformWriteTextAction
    | PerformSetCheckedAction
    | PerformSelectOptionsAction
    | PerformSubmitFormAction;
};
```

#### New error codes in protocol

Add to `ActionResultErrorCode`:

```ts
| 'stale_context'     // already exists (ADR 0036)
| 'page_navigated'     // new: page navigated since read
```

`target_not_found` remains unchanged (position out of range).

#### Content script lifecycle hooks

Add to content script:

```ts
// Increment page context version on navigation
window.addEventListener("pageshow", () => {
  pageContextVersion++;
});
```

## Alternatives Considered

### A. Hash-based element fingerprints

Attach a deterministic fingerprint (hash of tag name + classes + position in parent + text prefix) to each element during `read_current_page`. Content script re-derives fingerprints at action time and compares. **Rejected**: Fingerprints are fragile (style/class changes, A/B tests, i18n) and add complexity to the read response. Validated positional IDs are simpler and sufficient.

### B. Mandatory re-read before every action

Require agents to call `read_current_page` before each action, enforced by the MCP server via a context hash. **Rejected**: Doubles round-trips for sequential actions on a stable page. Validation fields are lighter-weight.

### C. Full DOM diff on stale detection

When a stale ID is detected, auto-re-read the page and return the fresh context along with the error. **Rejected**: Blurs read/action boundary; agent should explicitly decide whether to re-read. The error message tells them to re-read.

### D. `pageContextId` with SPA mutation detection

Use MutationObserver to detect significant SPA re-renders and increment `pageContextId`. **Rejected for initial implementation**: Too many false positives (ads, lazy images, tooltips) and false negatives (invisible data changes). Navigation (`pageshow`) is the clear-cut signal. SPA-level staleness is better handled by `stale_context` validation fields.

### E. No `pageContextId` — rely solely on `stale_context` for everything

Skip the navigation detection and only use per-field validation. **Rejected**: After navigation, the agent's entire context model is wrong (URL changed, title changed, all elements changed). Validating one field at a time doesn't convey "everything you know is wrong." `page_navigated` is a stronger, clearer signal.

## Consequences

**Positive:**

- All six action tools gain stale-context protection — no more silent wrong-button clicks or wrong-field fills
- `page_navigated` gives agents a definitive "reset your mental model" signal after navigation
- Clear error taxonomy (`stale_context` vs `page_navigated` vs `target_not_found`) enables targeted agent recovery strategies
- Fully backward compatible — all new fields are optional
- `pageContextId` is a lightweight integer, negligible payload impact

**Negative:**

- Every action tool schema gains optional fields — MCP clients need updates to use them
- `pageshow` event covers full navigations but not SPA re-renders — agents still need `stale_context` validation for those
- `expectedLabel` matching is fuzzy (substring, case-insensitive) and can produce false positives with very short labels like `"OK"` — but agents can pass more specific labels

**Neutral:**

- Protocol version unchanged — all additions are optional/backward compatible
- Content script gains a `pageContextVersion` module-scoped counter and a `pageshow` listener

## Implementation Plan

### Phase 1: Protocol types and error codes

1. Add `pageContextId` to `PerformActionRequest` in `packages/shared/src/protocol.ts`
2. Add `page_navigated` to `ActionResultErrorCode` union in `packages/shared/src/protocol.ts`
3. Add `expectedLabel` to `FormControlTarget` and `FormSubmitTarget` in protocol types
4. Add `expectedText` to `EditableActionTarget` in protocol types
5. Add `pageContextId` to `PageContextResponse` shape

### Phase 2: Content script — page context version

1. Add `pageContextVersion` counter in content script module scope
2. Increment on `pageshow` event
3. Include `pageContextId` in `read_current_page` response
4. Check `pageContextId` in action handlers, return `page_navigated` on mismatch

### Phase 3: Content script — validation for all action types

1. Add `expectedLabel` validation to form control target resolution (`fill_input`, `set_checked`, `select_options`)
2. Add `expectedText` validation to editable target resolution (`fill_editable`)
3. Add `expectedLabel` validation to form submit target resolution (`submit_form`)
4. Return `stale_context` with `detail` on validation failure

### Phase 4: Extension entry points

1. Chrome background script (`clients/extensions/chrome/src/background.ts`) — forward `pageContextId` and new validation fields (`expectedLabel` on `FormControlTarget`/`FormSubmitTarget`, `expectedText` on `EditableActionTarget`) in `perform_action` envelopes
2. Safari background entry (`clients/extensions/safari/src/background-entry.ts`) — same passthrough as Chrome for `pageContextId` and validation fields
3. Chrome content script entry (`clients/extensions/chrome/src/content.ts`) — register `pageshow` event listener to increment shared `pageContextVersion`
4. Safari content script entry (`clients/extensions/safari/src/content-script-entry.ts`) — register `pageshow` event listener to increment shared `pageContextVersion`

### Phase 5: MCP server — schema and tool updates

1. Add `pageContextId`, `expectedLabel`/`expectedText` to all action tool Zod schemas
2. Pass `pageContextId` through to WebSocket envelope
3. Pass validation fields through to content script
4. Map `page_navigated` error code in MCP result

### Phase 6: Tests

1. Unit tests in `content-handler.test.ts` for each action type's validation
2. Unit tests for `page_navigated` detection (pageContextId mismatch)
3. Integration tests for error propagation through MCP → WS → extension → content script
4. Negative tests: omitting validation fields preserves current behavior

### Phase 7: Documentation

1. Update ADR 0036 to note it is extended by ADR 0041
2. Update BrowserBridge skill guidance to recommend always passing validation fields
3. Document `page_navigated` recovery strategy for agent consumers
